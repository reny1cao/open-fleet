# Fleet v1.1 — Govern, Adapt, Remote

> Org Design Wizard + Collaboration Protocol + Agent Self-Management + Remote Servers

## Context

Fleet v1.0 shipped with Design (partial) + Instantiate layers. This spec completes the four-layer architecture:

| Layer | v1.0 | v1.1 |
|-------|------|------|
| **Design** | fleet.yaml has structure/topology, but not enforced | Org Design Wizard in SKILL.md — principles-driven team design |
| **Instantiate** | Done (init, start, stop, add-agent) | + Remote server support |
| **Govern** | None | Collaboration protocol + agent self-management |
| **Adapt** | None | Self-management foundation (lead can diagnose, suggest changes) |

## What Makes Fleet Different

Other multi-agent frameworks (CrewAI, AutoGen, MetaGPT) let you configure agents. Fleet **designs your organization** — it understands org structure principles native to AI agents and guides you through building an effective team. The SKILL.md is not a setup wizard, it's an org design consultant.

Key research findings driving this design:
- Context is the scarcest resource — agents degrade when their context window fills (Google research)
- Independent agents amplify errors 17.2x; centralized coordination limits to 4.4x
- Small teams (3-7) outperform larger ones for most tasks
- Adding agents often hits a ceiling — the right structure matters more than headcount
- Async Discord-native communication is Fleet's unique position vs synchronous frameworks

## 1. Org Design Wizard

### How it works

The wizard lives entirely in SKILL.md — no code changes needed. When a user starts Claude Code and invokes fleet's skill, Claude reads the org design principles and uses them to guide team design.

### Principles (embedded in SKILL.md)

These are the core knowledge that makes fleet an org design consultant, not just a process manager:

**P1: Context is the scarcest resource.**
The lead agent must never do heavy work — coding, file operations, long tool chains fill the context window and the agent forgets to respond. The lead coordinates only.

**P2: Small teams with clear boundaries.**
3-5 agents is the sweet spot. Beyond that, add a coordination layer. Each agent has one clear responsibility — no "full-stack" agents.

**P3: Every agent needs one clear job.**
Role confusion causes duplicated work or gaps. Each agent's identity must include: (a) what it does, (b) what it doesn't do, (c) what output format it produces, (d) who it delegates to.

**P4: Verification breaks error amplification.**
Without verification, errors compound across agents (17.2x amplification). Include a reviewer role for critical paths, or have the lead verify worker output before acting on it.

**P5: Remote machine = remote agent.**
If work needs to happen on a server, put an agent on that server. Don't SSH from local — the agent should have direct access.

**P6: Start small, add later.**
Begin with 2 agents (lead + worker). Add specialists when you feel the pain. `fleet add-agent` makes this frictionless.

### Quick-start templates

CLI provides templates as starting points (not required):

```bash
fleet init --template dev-team    # lead + coder + reviewer
fleet init --template research    # lead + researcher + analyst
fleet init --template ops         # lead + local-dev + remote-ops
```

Templates are YAML files in `~/.fleet/templates/`. Community can contribute new ones. Each template includes pre-configured roles, suggested team size, and a description of when to use it.

But templates are just shortcuts — the SKILL.md principles are the real product. Claude Code reads them and can design a custom team for any use case.

### SKILL.md flow update

Step 3 (currently "Understand user needs and design the team") becomes the heart of the experience:

1. Ask what the user wants to accomplish
2. Ask what machines they have (local only? remote servers?)
3. Based on principles P1-P6, suggest a team:
   - Who's the lead? (P1: lead never does heavy work)
   - How many workers? (P2: start small, 2-3)
   - Need a reviewer? (P4: critical paths need verification)
   - Remote agents? (P5: remote work = remote agent)
4. Explain the reasoning — "I suggest a reviewer because P4 says verification breaks error amplification..."
5. User confirms or adjusts
6. **Emit the CLI command** — wizard produces the exact `fleet init` command:
   ```
   fleet init --token T1 --token T2 --name my-team --agent lead:local:lead --agent coder:local:worker --agent reviewer:local:reviewer
   ```
   Or if user prefers a template: `fleet init --template dev-team --token T1 --token T2`
7. Proceed to Step 4 (token creation) in SKILL.md

**Template vs wizard flow:** If user says something that maps to a known template ("I want help coding"), offer the template as a starting point: "Here's a common pattern for dev teams — Lead + Coder + Reviewer. Want to start from this, or customize?" If the user's use case doesn't fit any template, fall back to P6: "Start with lead + 1 worker, expand when you feel the pain."

**Example of a well-formed agent identity (P3):**
```
You are **coder**, a worker in the fleet. Bot ID `123`.

## What you do
Write code, fix bugs, implement features. You receive tasks from the lead with specific requirements.

## What you don't do
Don't review your own code — that's reviewer's job. Don't deploy — that's ops. Don't talk to the user directly — route through lead.

## Output format
When you complete a task, reply with: what you changed, what files were touched, and whether tests pass.

## Who you delegate to
If you need code review: @mention reviewer. If you need deployment: @mention ops. If you're stuck: @mention lead.
```

This template shapes how the wizard crafts identities. The SKILL.md includes this as a reference pattern.

## 2. Collaboration Protocol

### Problem

Right now agents receive a message and silently start working. The user (and other agents) have no idea:
- Was the message received?
- Is the agent working on it?
- When will it be done?
- Did it fail?

### Solution: behavioral norms in identity

Written into every agent's identity.md at generation time. Not rigid templates — default norms that users can customize.

### Default norms

**Ack (all agents):**
When you receive an @mention or task, immediately react with an emoji or short reply ("On it", "Looking into this"). The sender needs to know you received the message. Never start working silently.

**Completion (all agents):**
When you finish a task, reply with the result. Be concise — conclusions first, details after. If you're replying to a specific message, use Discord's reply feature.

**Failure (all agents):**
If you can't do something, say so immediately. State what went wrong and what you need. Don't go silent on failure.

**Handoff (all agents):**
When you need another agent's help, @mention them directly. Without @mention, they won't receive the message. Include: what you need, what format you expect, any relevant context. Keep it concise — the receiving agent's context is limited.

### Implementation

These norms are added to the `buildIdentityPrompt()` function in `src/core/identity.ts`. They replace the current minimal "Rules" section with richer behavioral guidance.

The norms are **not** in CLAUDE.md (which is for dynamic roster data). They're in identity.md (system prompt) because behavioral norms should be stable and high-priority.

### Customization

Users can override norms by:
1. Editing `identities/<agent>.md` directly after generation
2. Defining custom norm sets in `identities/norms/` (like roles)

(`--norms` flag for init is deferred to v1.2 — manual editing is sufficient for v1.1)

## 3. Agent Self-Management

### Who manages?

Management ability follows the org structure — not a hardcoded role name:

- **Star topology** (`structure.lead: pm`) — the lead agent gets management knowledge
- **Hierarchy** (multiple managers) — each manager manages their sub-team
- **Mesh** (no lead) — every agent gets basic status/doctor ability

### What can the manager do?

**v1.1 (conservative — suggest + user approves):**
- `fleet status` — check who's online
- `fleet doctor` — diagnose issues
- `fleet inject <agent> <role>` — change a running agent's role
- Suggest: "Worker seems stuck, should I restart it?"
- Suggest: "We might need another agent for this workload"
- Does NOT autonomously add/remove agents or restructure

**Future (v1.2+ — autonomous within boundaries):**
- Auto-restart crashed agents
- Dynamic role assignment based on workload
- Propose org changes to user

### Global fleet config

Currently fleet commands need to find `fleet.yaml` via CWD. Fix:

`fleet init` writes the fleet directory path to `~/.fleet/config.json`:

```json
{ "defaultFleet": "/Users/renyitsao/workspace/qa" }
```

`findConfigDir()` in `src/core/config.ts` adds this as the last fallback:

```
$FLEET_CONFIG → CWD → $FLEET_DIR → ~/.fleet/config.json → error
```

This means any agent can run `fleet status` from any directory. No `cd` needed.

### Implementation

Management knowledge is injected into the identity based on the agent's position in the org structure:

```typescript
function isManager(agentName: string, config: FleetConfig): boolean {
  if (config.structure?.lead === agentName) return true
  // Future: check hierarchy for manager nodes
  return false
}
```

**Exact text injected into manager's identity.md:**

```markdown
## Fleet Management

You are the team coordinator. You can manage the fleet using these commands in your terminal:

### Check team status
`fleet status` — see who's online/offline
`fleet status --json` — machine-readable output

### Diagnose issues
`fleet doctor` — run all health checks
`fleet doctor --json` — machine-readable

### Change an agent's role
`fleet inject <agent> <role>` — hot-swap role without restart
Available roles: writer, reviewer, ops (or any in identities/roles/)

### Restart an agent
`fleet stop <agent>` then `fleet start <agent>`

### Your responsibilities
- When you receive a task from the user, decide: handle it yourself (simple) or delegate to a teammate
- If delegating, @mention the teammate with clear instructions
- If a teammate doesn't respond, check `fleet status` to see if they're online
- If something seems wrong, run `fleet doctor` to diagnose
- You may suggest adding agents or changing roles to the user, but don't execute without their approval
```

The fleet config directory path is read from `~/.fleet/config.json` and included in the identity so the agent doesn't need to `cd`:

```markdown
Fleet config directory: /Users/renyitsao/workspace/qa
```

## 4. Remote Server Support

### Architecture

Remote agents run as Claude Code sessions on remote machines via SSH + tmux. The local `fleet` CLI orchestrates everything — SSH in, start tmux, launch Claude Code.

### Prerequisites (remote machine)

- SSH access with key auth (host alias in `~/.ssh/config`)
- `bun` installed (for Claude Code Discord plugin)
- `claude` CLI installed and logged in
- `tmux` installed

`fleet doctor` checks all of these for configured remote servers.

### fleet.yaml config

```yaml
servers:
  singapore:
    ssh_host: sg-server    # matches ~/.ssh/config host alias
    user: dev

agents:
  ops:
    role: ops
    server: singapore
    workspace: ~/projects
```

### fleet start flow for remote agents

```typescript
// In start.ts:
const runtime = agent.server === "local"
  ? new TmuxLocal()
  : new TmuxRemote(config.servers[agent.server])
```

Everything else (identity generation, access.json, partner bot sync) stays the same — the files are generated locally then SCP'd to the remote.

### Trust prompt handling

Remote machines will hit the Claude Code trust prompt on first start (new workspace). The same polling loop from TmuxLocal works over SSH — just slower. Increase timeout for remote (15s instead of 10s).

## Config Validation

`loadConfig()` must validate that every agent's `server` field is either `"local"` or a key in `config.servers`. Invalid server names throw a typed config error at load time, not at runtime during `fleet start`.

## Success Criteria

- New user runs the SKILL.md flow → gets a well-designed team with clear reasoning
- Agents acknowledge messages immediately (no silent starts)
- Agents @mention teammates on handoff (no lost tasks)
- Lead agent can run `fleet status` and `fleet doctor` from any directory
- `fleet start ops --wait` starts an agent on a remote server via SSH
- `fleet doctor` validates remote server connectivity and prerequisites
- Community can contribute new org principles and templates by editing markdown/YAML files
